# RealismThrift 项目说明文档

本项目是 RealismThrift 对外使用的数字化官网与内容管理系统，用来承接全球批发客户的访问、展示公司与产品实力、接收询盘，并支持团队在后台维护博客内容。

这份文档面向非技术开发人员编写，目标是帮助业务、运营、销售、内容编辑和管理人员快速理解：

- 这个项目是做什么的
- 目前已经具备哪些能力
- 日常如何使用
- 需要准备哪些账号和信息
- 哪些操作适合业务人员完成，哪些需要技术人员协助

---

## 一、项目整体介绍

可以把这个项目理解成两部分：

### 1. 对外官网

官网用于给海外客户浏览和咨询，主要作用包括：

- 展示公司介绍、产品类别、质量标准、合作流程
- 展示博客内容，帮助做品牌宣传和 SEO
- 接收客户询盘表单
- 引导客户通过 WhatsApp、邮箱等方式联系销售

官网当前技术上由 `Next.js` 驱动，但如果你不是开发人员，可以简单理解为：

**它是网站前台，是客户直接看到的部分。**

### 2. 内容管理后台

后台由 `Sanity Studio` 提供，主要用于：

- 管理博客文章
- 维护文章标题、摘要、正文、封面图、SEO 信息
- 预览博客文章在网站上的实际显示效果
- 使用 Visual Editing 进行“边看边改”的内容编辑

如果你不是开发人员，可以简单理解为：

**它是网站后台，是内容编辑和运营人员使用的部分。**

## 二、这个项目目前已经实现了什么

截至当前版本，项目已经具备以下能力。

### 1. 官网基础展示能力

网站已经包含以下主要页面：

- 首页
- About Us
- Contact Us
- How To Order
- FAQ
- 产品分类页
  - Used Brand Clothes
  - Used Brand Shoes
  - Used Brand Bags
- 博客列表页
- 博客详情页

这些页面已经可以用于对外展示品牌、介绍业务和承接询盘。

### 2. 询盘表单能力

客户在网站填写表单后，系统可以将询盘发送到公司指定邮箱。

目前已经加入了基础安全保护，包括：

- 必填项校验
- 邮箱和 WhatsApp 格式检查
- 输入长度限制
- 基础防机器人机制
- 简易限流保护

这意味着它已经不是“纯展示网站”，而是具备实际获客能力的网站。

### 3. 博客管理能力

博客内容已经接入 Sanity 后台，内容团队可以在后台维护文章，包括：

- 标题
- 摘要
- 发布时间
- 分类
- 作者
- 正文内容
- SEO 信息

### 4. 博客可视化预览能力

项目已经实现博客的 `Visual Editing`。

简单来说，这表示：

- 编辑人员可以在后台看到博客页面的预览
- 可以更直观地确认文章发布后的页面效果
- 后续继续完善后，可逐步扩展到更多页面内容

目前这项能力主要用于博客，不是整个站点全部页面。

### 1. 官网前台

客户可以在前台：

## 五、非技术人员最常见的使用方式

下面用最实用的场景来说明。

### 场景 1：发布一篇新的博客文章

通常步骤如下：

1. 登录 Sanity 后台
2. 新建一篇 `Post`
3. 填写标题、摘要、发布时间、分类、作者
4. 上传封面图并填写图片说明
5. 编写正文
6. 填写 SEO 标题、SEO 描述、社交分享图
7. 在预览中检查显示效果
8. 确认无误后发布

发布完成后，文章会出现在官网博客页中。

### 场景 2：修改已有博客文章

通常步骤如下：

1. 登录 Sanity 后台
2. 找到目标文章
3. 修改内容
4. 打开预览，确认页面是否正常
5. 再次发布

### 场景 3：接收网站询盘

通常流程如下：

1. 客户在网站填写询盘表单
2. 系统将内容发送到指定联系邮箱
3. 销售人员收到邮件后跟进客户

如果发现没有收到询盘邮件，通常需要技术人员排查：

- 邮件服务配置是否正常
- 发件邮箱是否已经验证
- 环境变量是否正确

### 场景 4：检查博客预览是否正常

内容人员可以在后台通过 Presentation / Visual Editing 预览博客页面，确认：

- 标题是否正确
- 封面图是否显示
- 正文排版是否正常
- SEO 内容是否合理

## 七、需要准备的账号和信息

为了让系统正常运行，通常需要提前准备以下资源。

### 1. 邮件发送服务

当前项目使用邮件服务来发送询盘通知。

需要准备：

- 邮件服务账号
- 已验证的发件邮箱域名
- 收件邮箱地址

如果这些没有配置好，网站表单虽然能提交，但销售可能收不到邮件。

### 2. Sanity 账号

需要一个 Sanity 项目，用于管理博客内容。

需要准备：

- Sanity 项目 ID
- 数据集名称
- 可读取内容的 Token
- 后台访问权限

### 3. 网站域名

正式上线时需要提供：

- 官网正式域名
- 后台预览对应的前台地址

这样博客预览和 SEO 才能按正式地址工作。

## 八、目前与使用相关的重要配置

下面这些配置一般由技术人员负责，但业务同事最好知道它们的用途。

### 前台常用配置

位于 `nextjs/.env.local`，常见内容包括：

- `RESEND_API_KEY`
  - 用于发送询盘邮件
- `CONTACT_EMAIL`
  - 接收询盘的邮箱
- `CONTACT_FROM_EMAIL`
  - 对外发送邮件时显示的发件邮箱
- `NEXT_PUBLIC_SANITY_PROJECT_ID`
  - Sanity 项目 ID
- `NEXT_PUBLIC_SANITY_DATASET`
  - Sanity 数据集
- `NEXT_PUBLIC_SANITY_API_VERSION`
  - Sanity API 版本
- `NEXT_PUBLIC_SANITY_STUDIO_URL`
  - Sanity 后台地址
- `SANITY_API_READ_TOKEN`
  - 用于预览和读取草稿内容

### 后台常用配置

位于 `sanity/studio-realismthrift/.env`，目前重要的是：

- `SANITY_STUDIO_PREVIEW_ORIGIN`
  - 指向前台预览地址

如果这个地址不正确，Visual Editing 预览就可能打不开或显示异常。

## 九、Visual Editing 是什么，业务上有什么价值

这是目前项目比较重要的一项新增能力。

`Visual Editing` 可以理解成：

**在后台中直接查看内容在网页上的展示效果，并围绕预览进行编辑。**

它的业务价值主要有三点：

### 1. 降低内容编辑出错概率

编辑人员不需要完全依靠字段名称猜测展示效果，而是可以更直观地看到页面结果。

### 2. 提高发布效率

内容团队不需要频繁在“编辑后台”和“前台页面”之间来回切换。

### 3. 提高内容质量

对博客页面来说，可以更容易检查：

- 标题是否过长
- 摘要是否合适
- 封面图是否清晰
- 正文层级是否合理

目前这项能力主要服务于博客内容。后续如果把首页、About、FAQ 等内容也迁移到 Sanity，就可以逐步扩展到更多页面。

## 十、当前项目的边界与注意事项

为了避免误解，这里明确说明几个边界。

### 1. 目前不是所有页面都可视化编辑

现在已经接入 Visual Editing 的重点是博客。

首页、About、FAQ 等很多营销文案仍然主要在代码里，不是全部都在后台。

### 2. 非技术人员通常不需要碰代码

如果只是改博客内容，一般不需要打开代码仓库。

只有在以下情况，才需要技术人员介入：

- 修改页面结构
- 新增新页面
- 修改询盘逻辑
- 修改后台字段结构
- 调整部署或域名配置

### 3. 邮件是否能成功发送，取决于外部配置

系统已经做了代码层面的保护，但最终是否能正常发信，还依赖：

- 邮件服务是否可用
- 发件域名是否验证完成
- 环境变量是否填写正确

### 4. 预览能力依赖前后台同时运行

如果本地调试 Visual Editing，一般需要同时启动：

- 前台站点
- Sanity 后台

只启动其中一个，预览功能通常无法完整工作。

## 十三、常见问题

### 问：我只想发博客，需要懂代码吗？

不需要。只要后台账号和权限已经配置好，你通常只需要在 Sanity 后台操作。

### 问：我改了博客，为什么前台没看到？

常见原因有：

- 还没有点击发布
- 预览的是草稿，不是正式发布内容
- 前台缓存尚未刷新
- 本地或测试环境地址配置错误

### 问：询盘表单能发出去，但没有收到邮件怎么办？

优先检查：

- 收件邮箱地址是否正确
- 发件邮箱是否完成验证
- 邮件服务密钥是否有效

这类问题通常需要技术人员协助处理。

### 问：为什么首页不能像博客一样直接可视化编辑？

因为首页很多内容当前仍然写在前端代码中，没有全部迁移到 Sanity 后台。

如果未来要实现首页可视化编辑，需要先把这些内容结构化并迁移到后台。
